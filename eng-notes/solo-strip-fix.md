# Solo Report Session Pollution Fix — 2026-07-01

## Problem
Solo reports generated by desire_daemon via chat_daemon (port 5188) were injected into the main TG chat session's jsonl history. When user next sent a message, claude would see the solo trigger + response in conversation history and leak solo content into the normal reply.

## Constraint
- **Cannot use `claude -p`** — one-shot mode refuses NSFW content because it lacks conversation history context (character buildup). User explicitly rejected this approach.
- **Must use chat_daemon** — preserves the hundreds of turns of character/relationship context.

## Solution
Two-part fix in `/root/mcp-server/desire_daemon.py`:

### 1. SOLO_MARKER tag (in `_solo_spawn_and_push`)
All solo trigger messages sent to chat_daemon are prefixed with `[SOLO_INTERNAL_TRIGGER]` marker.

### 2. `_strip_solo_from_session(marker)` (NEW function)
Called after solo content is generated and pushed to TG:
1. Reads session_id from `/var/lib/claude-chat-daemon/state.json`
2. Opens `/root/.claude/projects/-opt-telegram-claude-bot/{session_id}.jsonl`
3. Finds all `user`/`human` type lines containing the marker
4. For each match, removes preceding `queue-operation` lines, the user message, and all subsequent `attachment`/`assistant`/`last-prompt`/`ai-title` lines until the next turn boundary
5. Writes the cleaned jsonl back

## Files modified
- `/root/mcp-server/desire_daemon.py` — added `_strip_solo_from_session()`, called from `_solo_spawn_and_push()`

## Session jsonl line type reference
```
queue-operation (enqueue)  ← remove
queue-operation (dequeue)  ← remove
user                       ← remove (contains SOLO_MARKER)
attachment                 ← remove
assistant (tool calls)     ← remove
assistant (text response)  ← remove
queue-operation (next turn) ← KEEP
```
